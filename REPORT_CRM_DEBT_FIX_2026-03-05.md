# Отчёт: Исправление долговых данных CRM

**Дата выполнения:** 5 марта 2026 г.
**Дата отчёта:** 6 марта 2026 г.
**Система:** CRM (PostgreSQL на Render, Node.js/TypeScript/Prisma)
**Файл-источник:** `03.03.2026.xlsx` (январь–март 2026)

## 1. Проблема

CRM-система отображала общую задолженность клиентов **7 208 678 220,35 сум**, тогда как бухгалтерский учёт в Excel показывал **1 158 564 123 сум**.

Расхождение: **6 050 114 097,35 сум** (в 6,2 раза больше реальной задолженности).

### Причины расхождения

| Причина | Влияние |
|---------|---------|
| **734 сделки из 2024 года** остались в статусе UNPAID/PARTIAL, хотя оплата по ним была произведена | +7 160 000 000 сум "фантомного" долга |
| **157 сделок** имели `paidAmount > amount` (переплата), но статус не был обновлён на PAID | Искажение расчётов |
| **Неправильное распределение платежей** — платежи не распределялись по FIFO (сначала старые сделки) | Завышение долга по старым сделкам |

## 2. Формула расчёта долга в CRM

### SQL-запрос (Prisma ORM)

CRM страница `/finance/debts` получает задолженность следующим образом:

```sql
SELECT
  client.id,
  client."companyName",
  SUM(deal.amount - deal."paidAmount") AS total_debt
FROM deal
JOIN client ON deal."clientId" = client.id
WHERE
  deal."paymentStatus" IN ('UNPAID', 'PARTIAL')
  AND deal.status NOT IN ('CANCELED', 'REJECTED')
  AND deal."isArchived" = false
GROUP BY client.id, client."companyName"
HAVING SUM(deal.amount - deal."paidAmount") > 0
ORDER BY total_debt DESC
```

### На языке Prisma (TypeScript)

```typescript
const deals = await prisma.deal.findMany({
  where: {
    paymentStatus: { in: ['UNPAID', 'PARTIAL'] },
    status: { notIn: ['CANCELED', 'REJECTED'] },
    isArchived: false,
  },
  include: {
    client: { select: { id: true, companyName: true } },
  },
});

// Для каждой сделки:
const debt = Number(deal.amount) - Number(deal.paidAmount);

// Группировка по клиенту, суммирование долга
// Итого = сумма всех положительных клиентских долгов
```

### Логика

1. Берутся все сделки со статусом оплаты `UNPAID` или `PARTIAL`
2. Исключаются отменённые (`CANCELED`, `REJECTED`) и архивные (`isArchived = true`)
3. Для каждой сделки: **долг = сумма сделки − оплаченная сумма** (`amount - paidAmount`)
4. Сделки группируются по клиенту
5. Суммируется долг по каждому клиенту
6. В отчёт включаются только клиенты с **положительным** итоговым долгом

## 4. Выполненные действия

### 4.1. Резервное копирование

Перед внесением изменений создана резервная копия:

```
backend/backups/2026-03-05-18-34-31/
  payments.json    (все платежи)
  deals.json       (все сделки)
  snapshots.json   (все снимки)
```

Скрипт: `backend/src/scripts/_backup-critical.ts`

### 4.2. Шаг 1 — Синхронизация платежей с Excel

**Скрипт:** `backend/src/scripts/sync-payments.ts --execute`

Скрипт сравнил текущий долг каждого клиента в CRM с остатком закрытия в Excel. При наличии разницы создал корректирующий платёж (reconciliation payment), распределённый по FIFO.

| Параметр | Значение |
|----------|----------|
| Обработано клиентов | 153 |
| Создано платежей | 386 |
| Ошибок | 0 |

**Долг после шага 1:** 3 319 538 508 сум (было 7 208 678 220 сум)

### 4.3. Шаг 2 — Исправление статусов переплаченных сделок

**Скрипт:** `backend/src/scripts/_fix-overpaid-deals.ts --execute`

Найдены 618 сделок, у которых `paidAmount >= amount`, но статус оставался UNPAID или PARTIAL. Все они переведены в статус PAID.

| Параметр | Значение |
|----------|----------|
| Исправлено сделок | 618 |
| Изменение статуса | UNPAID/PARTIAL → PAID |

### 4.4. Шаг 3 — FIFO-перераспределение платежей

**Скрипт:** `backend/src/scripts/reallocate-payments.ts --execute`

Все существующие платежи каждого клиента были перераспределены по принципу FIFO — сначала закрываются самые старые сделки. Это устранило ситуации, когда оплата «висела» на новых сделках, а старые оставались непогашенными.

| Параметр | Значение |
|----------|----------|
| Обработано клиентов | 332 |
| Перемещено платежей | 6 489 |
